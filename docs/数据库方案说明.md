# DrumPractise 数据库方案说明

本文档描述 DrumPractise 当前的数据库**表结构、写入/查询边界、以及后续扩展策略**。文中路径均以仓库根目录 `DrumPractise/` 为基准。

---

## 1. 目标与范围

- **目标**：让数据库在功能扩展时尽量避免频繁结构变更，同时为后续统计模块提供稳定的数据来源。
- **首版业务约束**：
  - **事件粒度**：仅记录「每轮练习结束」。
  - **筛选场景**：首版统计仅按 `subject`（练习类型）+ 时间范围筛选/汇总；不按 BPM/模式等字段做 SQL 级筛选。
- **代码边界（主要文件）**：
  - SQLDelight 表与查询：`composeApp/src/commonMain/sqldelight/com/drumpractise/db/PracticeEvent.sq`
  - 埋点写入：`composeApp/src/commonMain/kotlin/com/drumpractise/app/analytics/PracticeAnalytics.kt`
  - 统计仓库（读写边界）：`composeApp/src/commonMain/kotlin/com/drumpractise/app/analytics/AnalyticsRepository.kt`
  - 数据库单例：`composeApp/src/commonMain/kotlin/com/drumpractise/app/data/DrumDatabaseSingleton.kt`
  - 多平台打开数据库：`composeApp/src/*Main/kotlin/com/drumpractise/app/data/DrumDatabaseProvider.*.kt`

---

## 2. 数据模型：单一事件表（append-only）

数据库采用单表事件模型：每次练习完成写入一行，后续统计以扫描/聚合为主。

### 2.1 表：`practice_event`

定义见：`composeApp/src/commonMain/sqldelight/com/drumpractise/db/PracticeEvent.sq`。

- **固定列**：
  - `id`：自增主键。
  - `created_at`：事件发生时间（epoch millis）。
  - `local_user_id`：本地用户标识（来自 `AppSettings`）。
  - `event_type`：首版固定为 `PRACTICE_ROUND_COMPLETED`。
  - `subject`：练习类型（如 `SEPARATION`、`ACCENT_SHIFT`），首版唯一用于 SQL 级业务筛选的字段。
  - `payload_schema`：payload 版本标识（例如 `SeparationPracticeState@v1`），用于将来 payload 演进时的读侧兼容。
  - `payload_json`：完整练习状态的 JSON 快照（真源）。
  - `app_version` / `app_build`：可选，首版允许为空，主要用于统计排错与版本对账。

### 2.2 索引

首版只优化两类查询：

- **按时间范围**：例如最近一周内所有事件。
- **按 subject + 时间范围**：例如最近一周内 `SEPARATION` 的事件。

因此在 `practice_event` 上建立了：

- `created_at DESC` 索引
- `(subject, created_at DESC)` 联合索引

## 3. 写入与查询边界

### 3.1 写入：练习完成事件

写入入口：`PracticeAnalytics.record*PracticeRound(...)`。

- 事件类型固定为：`PRACTICE_ROUND_COMPLETED`
- `subject` 来自练习模块（`SEPARATION` / `ACCENT_SHIFT`）
- `payload_json` 为对应 `PracticeState` 的序列化 JSON
- `payload_schema` 用于区分未来版本（例如 `@v2`）

底层写入通过 `AnalyticsRepository.appendPracticeRoundCompleted(...)` 统一落库。

### 3.2 查询：统计模块首版只按 subject

查询入口：`AnalyticsRepository`。

- `countPracticeRoundsBySubjectBetween(subject, from, to)`
- `listPracticeRoundsBySubjectBetween(subject, from, to)`

> 若未来出现「需要按 BPM/模式等做 SQL 级筛选」的需求，首版不会提前在表中加入派生列；应通过单独评审决定：增加可筛列（migration）或增加投影表（可重建）两条路径之一。

---

## 4. 演进策略（避免频繁改表）

- **优先演进 JSON**：新增字段、调整结构优先落在 `payload_json` 内，并通过 `payload_schema` 区分版本；读侧用 `ignoreUnknownKeys` 兼容旧数据。
- **首版不做“派生列”**：BPM/模式/level 等只存在于 JSON，避免为了统计筛选而频繁新增列。
- **需要 SQL 级筛选时再引入结构**：
  - 选项 A：在 `practice_event` 增加少数可筛列并创建索引（需要 migration）。
  - 选项 B：引入可重建的投影表（例如 `practice_event_index` 或 `stats_*`），允许 `DROP`/重建以降低对主表的影响。

---

## 5. 数据库实例与重建策略

- **数据库单例**：`DrumDatabaseSingleton.instance` 是应用内唯一入口，避免多处打开同一个 SQLite 文件。
- **完全重建（无历史迁移）**：本次方案假设老版本无人安装，因此数据库文件名改为 `drum_v2.db`，新版本直接使用空库启动，无需旧数据迁移。
